# pi-weather Deployment

How `pi-weather` is deployed, run, and kept alive across reboots on the two
Raspberry Pis.

## Hosts

| Host | DNS | IP | App location |
|------|-----|----|--------------|
| pi1 | pi1.anselbrandt.net | 192.168.8.144 | `/home/ansel/dev/pi-weather` |
| pi2 | pi2.anselbrandt.net | 192.168.8.138 | `/home/ansel/dev/pi-weather` |

SSH aliases `ssh pi1` / `ssh pi2` are configured and resolve to the hosts above.
Both use key-based auth (no password).

## What runs

- `weather.js` fetches current Montreal weather from the `weather.gc.ca` JSON API
  and renders temperature + a rain indicator to the Sense HAT 8x8 LED matrix.
- It refreshes every 60 seconds (interval lives inside `weather.js`).
- `getWeather.js` is a standalone API test script — **not** run in production.

## Runtime

- **Node.js 18.20.8**, installed via **nvm**. Built-in `fetch` is used (no
  `node-fetch`).
- Only npm dependency is `sense-hat-led` (see `package.json`).
- Over SSH you must `source ~/.nvm/nvm.sh` before any `nvm` / `node` / `pm2`
  command, otherwise they aren't on `PATH`.

## Process management — PM2 (not cron)

The app runs as a **PM2** process named `weather` (fork mode). There is **no
crontab entry** — scheduling/refresh is handled inside the script, and process
supervision is handled by PM2.

| Field | Value |
|-------|-------|
| Process name | `weather` (id 0) |
| Script | `/home/ansel/dev/pi-weather/weather.js` |
| Interpreter | `node` (v18.20.8) |
| cwd | `/home/ansel/dev/pi-weather` |
| Mode | fork |

Useful commands (after `source ~/.nvm/nvm.sh`):

```bash
pm2 list              # status of all processes
pm2 logs weather      # tail logs
pm2 restart weather   # restart after a code change
pm2 describe weather  # full process details
```

## Survives reboot — systemd + pm2 resurrect

PM2 itself does not start on boot. Reboot survival is provided by a **systemd
unit** that was generated by `pm2 startup`:

- Unit: `pm2-ansel.service` — **enabled** on both Pis.
- `ExecStart=.../pm2 resurrect` (runs as user `ansel`).
- On boot, systemd starts the PM2 daemon and runs `pm2 resurrect`, which
  relaunches every process saved in `~/.pm2/dump.pm2`.

The dump file is the snapshot restored on every boot. It is written by
`pm2 save` — so **after changing what PM2 runs, you must `pm2 save`** or the
change will be lost on the next reboot.

```bash
pm2 save              # persist current process list to ~/.pm2/dump.pm2
systemctl is-enabled pm2-ansel   # should print "enabled"
```

## Deploy a change

1. Commit and push locally:
   ```bash
   git commit -am "..."
   git push origin master
   ```
2. On each Pi (`ssh pi1`, then `ssh pi2`):
   ```bash
   cd ~/dev/pi-weather
   git pull
   source ~/.nvm/nvm.sh
   pm2 restart weather
   ```
3. If you changed *how* the process is launched (script path, name, args), also
   run `pm2 save` so the reboot snapshot stays in sync.
